abstract: 面向智能体工作负载的批量推理持续且累积地加大GPU键值（KV）缓存压力，常在内存容量耗尽之前就引发严重的吞吐量下降。我们将此现象定义为“中期抖动”，这是一种此前未被充分表征的病理状态，即随着长期存活的智能体随时间积累状态，缓存效率会骤降。我们认为，缓解这一病理需要超越反应式的请求级缓存管理，迈向主动的智能体级准入控制。受分布式系统拥塞控制的启发，我们将KV缓存视为一种共享资源，其高效利用依赖于反馈驱动的调节。基于这一洞察，我们提出了CONCUR，一种轻量级控制层，它通过调节智能体准入来限制整体缓存压力，同时保持执行连续性。CONCUR采用一种缓存感知的控制算法，利用运行时缓存信号动态调整活跃智能体的数量。在大型模型和真实世界智能体工作负载上，CONCUR可防止中期抖动，并将Qwen3-32B和DeepSeekV3上的批量推理吞吐量分别提升至4.09倍和1.90倍，且与现有LLM推理服务系统保持兼容。

这是一篇面向**大模型智能体（Agent）批量推理场景**的系统优化论文。论文首次定义并刻画了该场景下的核心性能瓶颈“中期颠簸（Middle-Phase Thrashing）”**，并受网络拥塞控制启发，提出了一种轻量级、非侵入式的**Agent级主动准入控制中间件 `CONCUR`**。

以下从六个维度对论文进行详细拆解：

---

### 📖 一、 论文基本信息

- **标题**：`CONCUR: Proactive Agent-Level Admission Control for Efficient Agentic Batch Inference`
    
- **核心目标**：解决多Agent并发推理时GPU KV Cache效率崩溃问题，提升端到端吞吐量。
    
- **适用场景**：离线批量Agent推理（如Agentic RL rollout、数据蒸馏、大规模Agent评测等）。
    
- **关键成果**：在Qwen3-32B上最高提速 `4.09×`，在DeepSeek-V3上最高提速 `1.90×`，且兼容现有LLM服务引擎（如SGLang）。
    

---

### 🔍 二、 研究背景与核心问题

#### 1. Agent推理的内存特性

现代Agent多采用ReAct范式，在“推理-调用工具-观察”循环中不断追加上下文。这导致：

- 每个Agent的上下文长度与KV Cache占用**随时间单调递增**。
    
- 多个Agent**异步执行**：部分Agent在生成token，部分因等待外部工具而暂停。
    

#### 2. 现有系统的失效机制

主流LLM服务引擎（如vLLM、SGLang）采用**请求级（Request-Level）调度 + LRU淘汰 + CPU卸载**。但在Agent场景下：

- 暂停Agent的KV Cache因“最近未使用”被LRU激进淘汰。
    
- Agent恢复时，系统必须通过**重计算（Prefill Recomputation）** 或**PCIe回传**重建上下文，延迟随历史长度平方级增长。
    
- 高并发下PCIe带宽争抢进一步放大延迟。
    

#### 3. 核心发现：中期颠簸（Middle-Phase Thrashing）

作者通过真实负载追踪，发现Agent批量推理呈现典型的**三阶段演化**：

| 阶段                | 特征                     | 系统状态                                              |
| ----------------- | ---------------------- | ------------------------------------------------- |
| **预热期**           | 上下文短、前缀共享率高            | Cache命中率~90%，吞吐随并发单调上升                            |
| **中期颠簸**（占>90%时间） | 显存持续饱和(80~100%)，但命中率骤降 | 陷入`淘汰→重计算→再淘汰`死循环，重计算开销占总延迟49.1%。**此时增加并发反而降低吞吐** |
| **冷却期**           | 部分Agent结束释放显存          | 压力缓解，命中率部分恢复                                      |

💡 **关键结论**：瓶颈不是显存容量耗尽，而是**反应式请求级调度**与**Agent长生命周期状态连续性**之间的抽象错配。

---

### 💡 三、 核心洞察与设计哲学

1. **范式转变**：从 `反应式缓存管理（淘汰/卸载）` → `主动式Agent级准入控制`。以Agent为调度单元，而非单次生成请求。
    
2. **跨领域类比**：将GPU KV Cache类比为**网络带宽**，将并发Agent类比为**数据流**。过度提交会导致“拥塞崩溃”，需通过反馈闭环动态限流。
    
3. **非侵入式架构**：不替换现有推理引擎，而是在Agent框架与LLM Serving Engine之间插入轻量级控制层。
    

---

### ⚙️ 四、 CONCUR 系统设计

#### 1. 系统架构与执行流

Agent执行层 → CONCUR控制器 → LLM推理引擎(SGLang等) → GPU KV Cache

四阶段闭环：

1. **准入决策**：Agent提交生成请求至控制器，控制器决定`Admit`或`Pause`。
    
2. **批量生成**：准入Agent进入引擎进行Batched Decoding。
    
3. **工具挂起**：调用工具的Agent暂停，但控制器保护其逻辑状态不被淘汰。
    
4. **反馈调节**：控制器实时采集`KV Cache使用率(Ut)`与`命中率(Ht)`，动态调整并发窗口。
    

#### 2. 缓存感知准入控制算法（AIMD改进版）

借鉴TCP拥塞控制的`加性增/乘性减（AIMD）`，定义并发窗口 W_t（允许同时活跃的Agent数）：

$$  
W_{t+1}= \begin{cases} W_t + \alpha & \text{if } U_t < U_{low} \quad \text{(线性探测，提升利用率)} \\ W_t \times \beta & \text{if } U_t > U_{high} \land H_t < H_{thresh} \quad \text{(乘性骤降，快速退出颠簸)} \\ W_t & \text{otherwise} \quad \text{(稳定维持)} \end{cases}  
$$

  

- **信号设计**：U_t（使用率）作主动拥塞预警，H_t（命中率）作失效确认。双信号避免误触发。
    
- **缓冲区设计**：U_{low} 与 U_{high} 之间留有空隙，防止长上下文Agent突增内存引发震荡。
    
- **默认参数**：\alpha=2, \beta=0.5, U_{low}=0.2, U_{high}=0.5, H_{thresh}=0.2。
    
|**控制原语**：| 原语                 | 核心含义                                                     | 触发条件与约束                                               |
| :------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **`admit`（准入）**  | 授权Agent执行下一步生成请求，允许其进入LLM Serving Engine参与Batched Decoding，并正常使用GPU KV Cache。 | 当控制器判定当前缓存压力较低（$U_t < U_{low}$）或命中率健康时，从Pending队列中放行Agent。 |
| **`pause`（暂停）**  | 临时阻止Agent继续提交生成请求，但**完整保留其执行状态与KV Cache逻辑关联**（防止被LRU激进淘汰）。 | **严格仅在安全边界触发**（如单步生成结束、工具调用前后），绝不中断正在GPU上执行的 `in-flight computation`。用于显存压力逼近阈值时主动限流。 |
| **`resume`（恢复）** | 当系统资源水位回落至安全区间时，重新激活被 `pause` 的Agent，使其继续后续推理或工具交互。 | 由控制器协调唤醒节奏，避免瞬间大量恢复引发二次缓存拥塞（Renewed Cache Congestion）。 |，仅在生成步与工具调用边界操作，不中断飞行中计算。

### 📊 五、 实验评估与关键结果

**实验设置**：NVIDIA H100 (80GB) × 多卡，对比基线：

- `SGLang`（原生LRU）
    
- `SGLang + Request Control`（固定请求级限流）
    
- `SGLang + HiCache`（激进CPU卸载保缓存）
    

| 研究问题               | 核心发现                                                                                                                          |
| ------------------ | ----------------------------------------------------------------------------------------------------------------------------- |
| **Q1: 端到端吞吐**      | CONCUR全面最优。Qwen3-32B (Batch=256, TP=2) 延迟从2527s降至846s (**4.09×**)；DeepSeek-V3 (Batch=40) 提速 **1.90×**。基线在高并发下因重计算或PCIe争抢性能暴跌。 |
| **Q2: KV Cache行为** | CONCUR在中期维持高命中率（DeepSeek-V3 Batch=40时命中率73.36%，基线仅35.41%）。时序曲线显示CONCUR有效压制了命中率断崖式下跌，且避免过度CPU卸载。                               |
| **Q3: 静态 vs 动态限流** | 固定并发数（30/64/128）要么浪费显存，要么触发颠簸。CONCUR自适应调节，比最优固定配置快1.5~2.9×，证明**动态反馈控制不可或缺**。                                                  |
| **参数敏感性**          | U_{high} 在0.5~0.6较鲁棒；U_{low} 需精细校准（0.2最优），过低导致窗口无法增长，过高引发过载。                                                                  |

### 🎯 六、 总结与学术/工程价值

1. **理论贡献**：首次形式化定义 `Middle-Phase Thrashing`，揭示Agent批量推理中“显存未耗尽但效率先崩溃”的反直觉现象。
    
2. **方法创新**：将网络拥塞控制（AIMD）成功迁移至AI系统资源调度，提出以Agent为粒度的主动准入控制范式。
    
3. **工程落地**：CONCUR作为轻量中间件，无需修改底层Attention或PagedAttention实现，可即插即用于SGLang/vLLM等主流引擎，对Agentic RL、数据合成、大规模评测等高吞吐场景具有直接应用价值。
    
4. **局限与展望**：阈值参数仍需一定场景调优；未深入探讨跨节点分布式KV池或异构显存环境；未来可结合Agent执行图预测实现更细粒度的内存预留。
